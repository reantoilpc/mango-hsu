## Problem Statement

mango-hsu V2 已上線、V3 LIFF 通知已 ship，但**系統還沒對外公開**——因為 V2 的庫存模型只追蹤 `available: boolean`，去年（V1 線下實體交易）真實發生過兩種超賣情境：

1. **季末搶購**：同時下訂芒果乾，但量已不夠分配（人工處理：「慢的人拿少」）
2. **跨管道無同步**：自己接訂單時不知道姑姑那邊也接了訂單，導致超賣

V4 的目標是在系統公開前，把當初刻意省略的庫存模型補回來，並把姑姑這條線下管道納入系統，避免情境 2 重演。

## What Makes This Cool

不是「酷」，是「不丟臉」——系統公開後出現超賣是難堪的事，特別是給家人用的工具。V4 真正讓 mango-hsu 有資格說「對外接單」。

## Constraints

- 家庭規模流量：產季 < 100 單/天，非產季 ~0
- Cloudflare Workers free tier（10ms CPU、100k req/day）
- D1 SQLite（沒有 `SELECT ... FOR UPDATE`，但有 atomic UPDATE + batch transaction）
- 姑姑（家人）線下管道**不會**消失，永遠用 LINE 接單
- 姑姑科技舒適度：會用 LINE，**不會**登入後台或學新介面
- 同步精度只需到「當天」——線下/線上的 cross-channel race 不需即時解
- V4 跟系統公開一起上線，所以 schema/API/UX 都可以動，**不用顧 backwards compat**

## Premises（user 已確認）

1. **庫存單一真相**：客人單 + 姑姑單必須扣同一個 stock 數字。沒有 fair allocation、沒有 reservation pool。賣完就 SOLD OUT。
2. **姑姑端零變動**：她繼續用 LINE 接單，由 user 或太太當天在 admin 後台代建。
3. **同步精度=當天**：跨管道 race condition 不需解；只需解線上多客人之間的 race（D1 atomic UPDATE 即可）。
4. **接受的 fallback flow**：晚上對帳發現姑姑那邊接的單超出剩餘庫存 → 通知姑姑 → 姑姑回覆她的客人「只能出 X 包」。是設計接受的結果，不是 bug。
5. **V4 = 純最小修改**：stock 欄位 + atomic 扣減 + admin 代建單 + 改/刪 stock 還原。其他延伸（補貨通知、客人取消、預訂池、季末分配）一律**不做**。

## Approaches Considered

### Approach A — Minimal（共用客人 endpoint）
~200 行新 code、1 migration。客人 endpoint 為 admin bypass 開後門（auth/token/PDPA 分支），會逐步劣化。

### Approach B — Separation（分離 admin endpoint）⭐ Chosen
~280 行新 code、1 migration、1 個新 endpoint。客人 endpoint 維持乾淨；admin 路徑獨立、`source` 欄位給未來 dashboard 留分類門。

### Approach C — Reservation 短窗
~450+ 行新 code。對家庭規模過度工程，且跟 premise #4 衝突——已接受 fallback，不需 reservation 防護。

## Recommended Approach: B — 分離 admin endpoint

### Schema 改動

新 migration（`drizzle/000X_v4_stock.sql`）：

```sql
ALTER TABLE products ADD COLUMN stock INTEGER NOT NULL DEFAULT 0;
ALTER TABLE orders ADD COLUMN cancelled_at TEXT;
```

對應 `src/db/schema.ts` Drizzle 同步：

```ts
export const products = sqliteTable("products", {
  // ... 原欄位 ...
  stock: integer("stock").notNull().default(0),
});

export const orders = sqliteTable("orders", {
  // ... 原欄位 ...
  cancelled_at: text("cancelled_at"),
}, ...);
```

**為什麼沒有 `source` 欄位**（codex F7 catch）：原本想加 `source: 'web' | 'admin'` 區分客人/admin 訂單來源，但實際**唯一的用途**是「admin 代建單不送 Telegram」——這在 admin endpoint 直接不 call `notifyOrder()` 就夠了，不需要 column 持久化。要查「哪些單是 admin 代建」就 `JOIN audit_log WHERE action='admin_order_created'`，audit_log 已經是 single source of truth。

**為什麼 admin 代建單 `pdpa_accepted = 0`**（codex F8 catch）：原本想強制 1，但這會污染 PDPA 欄位語意（變成「admin 猜測 LINE 上有講」）。實際上：
- `pdpa_accepted = 1` 的明確意義是「客人在 form 上勾選了 PDPA 同意 checkbox」
- admin 代建單客人沒勾過 → `pdpa_accepted = 0` 才是真實
- Schema 既有 `pdpa_accepted INTEGER NOT NULL` 接受 0 / 1，不需 migration 改 NOT NULL
- 法律意義：Taiwan PDPA 接受口頭告知（姑姑 LINE 對話），但 schema 不該假裝書面同意已收到

**為什麼沒有 `cancelled_by`**：`audit_log` 在取消時會寫一行 `action='order_cancelled'` 含 `user_email` + `order_id`——已經是 single source of truth 記錄誰取消。要查「誰取消這單」就 `JOIN audit_log`。

**`available` 欄位保留**——admin 還是要能「下架某 SKU」即使有庫存（例如 SKU 暫時停售）。下單流程加雙重檢查：先 `available === true`，再 `tryDecrementStock`。

**SKU 第一次設 stock**：手動 `wrangler d1 execute mango-hsu-stage --command "UPDATE products SET stock=200 WHERE sku='dry-mango-200g'"`。**不**寫成 migration——數字是業務決定，不該寫死在 migration。

### Stock 扣減 helper（新 `src/lib/stock.ts`）

設計上分兩層：**底層回傳 prepared statements**（讓 caller 可以塞進自己的 batch，配合其他 SQL 一起 atomic commit），**上層包成 standalone async function**（給「只動 stock、不需要連動 INSERT/UPDATE 其他表」的場景用）。

這樣可以避免 D1 batch 不能 nested 的限制（cancel/edit-quantity 流程要把 stock 改動跟 `UPDATE orders` / `UPDATE order_items` / `INSERT audit_log` 包在同一個 batch 裡）。

```ts
import type { AppEnv } from "../db/client";

type Item = { sku: string; qty: number };

/** 底層：回傳 stock 扣減的 prepared statements。Caller 自己組 batch。 */
export function stockDecrementStmts(env: AppEnv, items: Item[]) {
  return items.map((it) =>
    env.DB.prepare(
      `UPDATE products SET stock = stock - ?1 WHERE sku = ?2 AND stock >= ?1`,
    ).bind(it.qty, it.sku),
  );
}

/** 底層：回傳 stock 還原的 prepared statements。Caller 自己組 batch。 */
export function stockRestoreStmts(env: AppEnv, items: Item[]) {
  return items.map((it) =>
    env.DB.prepare(`UPDATE products SET stock = stock + ?1 WHERE sku = ?2`)
      .bind(it.qty, it.sku),
  );
}

/**
 * 上層 standalone：atomic reserve + post-batch CAS 檢查 + 自動補償。
 * 給「下單流程」用（先確認 stock，再進去 INSERT order — 兩個 batch 分開）。
 * 不能跟其他 SQL 包進同一個 batch（因為它要看 results 才決定要不要 restore）。
 */
export async function tryDecrementStock(
  env: AppEnv,
  items: Item[],
): Promise<{ ok: true } | { ok: false; sold_out_sku: string }> {
  if (items.length === 0) return { ok: true };
  const stmts = stockDecrementStmts(env, items);
  const results = await env.DB.batch(stmts);
  for (let i = 0; i < results.length; i++) {
    if ((results[i].meta.changes ?? 0) === 0) {
      // 補償：restore 之前已扣的 SKU
      if (i > 0) {
        await env.DB.batch(stockRestoreStmts(env, items.slice(0, i)));
      }
      return { ok: false, sold_out_sku: items[i]!.sku };
    }
  }
  return { ok: true };
}

/**
 * 上層 standalone：純還原。給「下單失敗 unwind」用（INSERT 失敗、需要把 reservation 退回）。
 * 不需 CAS，single batch 內全部 commit。
 */
export async function restoreStock(env: AppEnv, items: Item[]): Promise<void> {
  if (items.length === 0) return;
  await env.DB.batch(stockRestoreStmts(env, items));
}
```

**使用模式總覽**：

| 場景 | 用什麼 | 為什麼 |
|---|---|---|
| 客人/admin 下單流程 | `tryDecrementStock` (上層) | 需要 post-batch 看 results 決定 SOLD_OUT，本質是 2-batch（reserve + insert）所以 helper 自己是 atomic |
| 下單 INSERT 失敗 unwind | `restoreStock` (上層) | 純還原，不需 CAS |
| Cancel 訂單 | `stockRestoreStmts` (底層) | 要跟 `UPDATE orders SET cancelled_at` + `INSERT audit_log` 包進**同一個 batch** atomic commit |
| Edit-quantity 增加 | `tryDecrementStock`（先）+ 然後另一個 batch 改 order_items + audit | 增加要 CAS（可能 SOLD_OUT），所以無法跟 `UPDATE order_items` 同 batch |
| Edit-quantity 減少 | `stockRestoreStmts` (底層) | 不需 CAS，可跟 `UPDATE order_items` + audit 同 batch |

**並發語意說明（重要——atomicity 不是免死金牌）**：

D1 `db.batch()` 在單 batch 內是 atomic（all-or-nothing transaction）。但 atomicity **只保證「整批 commit 或整批 rollback」**——它**不保證**「batch 內每個 UPDATE 的 affected rows 都 > 0」。

也就是說，一個 batch 裡 5 個 `UPDATE ... WHERE stock >= ?` 會全部執行並 commit，即使其中 3 個的 WHERE 條件不滿足、affected rows = 0。後者不是錯誤，是**成功但沒改任何 row**。

所以正確性是靠**兩個機制疊起來**：
1. 每個 UPDATE 的 `WHERE stock >= ?1` 是 CAS（compare-and-swap）：只在 stock 足夠時才扣
2. Batch 完成後我們**掃 results.meta.changes**，發現有任何 0-changes 的 row → 啟動「補償還原」（restore 之前已扣的 SKU），回 SOLD_OUT

兩個請求同時搶最後一包：兩條 UPDATE 都 `commit`，但只有 1 個 affected rows = 1，另一個 = 0 → 後者觸發補償，restore 自己之前已扣的，回 SOLD_OUT。**Net stock 一致正確**。

**Multi-SKU basket 的 transient phantom decrement**（codex F4）：客人下單 [SKU1=1, SKU2=1, SKU3=1]，SKU3 stock=0。Phase A batch commit 後 SKU1 / SKU2 各 stock-1（commit 了），SKU3 affected=0。我們發現後跑 Phase B compensating restore SKU1+SKU2。在 Phase A commit 與 Phase B commit 之間（微秒級），其他並發查詢若看 SKU1/SKU2 會看到「短暫被扣」的數字——可能造成另一客人短暫看到 SOLD_OUT（但下一秒 stock 又回升）。對 V4 沒有客人前端「剩 N 包」顯示（延後 V4.1）來說，這個 transient state 不可觀察、無實質影響。文件記錄起來，未來真要做「即時 stock 顯示」時再考慮 single-batch 重構。

### 客人 endpoint 改動（`src/pages/api/orders.ts`）

在現有 step 3（Product + availability + price snapshot）之後、step 4（Insert）之前插入：

### Admin endpoint（新 `src/pages/api/admin/orders.ts`）

```ts
import type { APIRoute } from "astro";
import { eq } from "drizzle-orm";
import { makeDb } from "../../../db/client";
import { orders, order_items, products, audit_log } from "../../../db/schema";
import { nextOrderId } from "../../../lib/order-id";
import { tryDecrementStock, restoreStock } from "../../../lib/stock";
import { expectedMemoFor, shippingFor } from "../../../lib/order-response";
import { env } from "../../../lib/env";
import { requireSameOrigin } from "../../../lib/csrf";

interface AdminOrderRequest {
  idempotency_key: string;  // 前端 form render 時 crypto.randomUUID() 產生，hidden input 帶過來
  name: string;
  phone: string;
  address: string;
  notes?: string;
  items: Array<{ sku: string; qty: number }>;
  // 沒 token、honeypot、pdpa_accepted
}

export const POST: APIRoute = async ({ request, locals }) => {
  // 1) CSRF（cheap，無 DB）先檢
  if (!requireSameOrigin(request)) {
    return new Response(JSON.stringify({ ok: false, error_code: "CSRF" }), { status: 403 });
  }
  // 2) Auth — middleware 已把 session 注入 locals（V2 既有機制）
  //    SessionInfo = { email: string; role: "admin" | "operator" } (src/lib/auth.ts)
  const session = locals.session;
  if (!session) {
    return new Response(JSON.stringify({ ok: false, error_code: "AUTH" }), { status: 401 });
  }
  // admin 與 operator 都可以代建單（兩者都是被信任的家人）。
  // 如果未來想限制只有 admin 能代建，改這條 if。
  if (session.role !== "admin" && session.role !== "operator") {
    return new Response(JSON.stringify({ ok: false, error_code: "FORBIDDEN" }), { status: 403 });
  }

  // 3) Validate（同客人 validate 但拿掉 PDPA / token 檢查；helper 抽到 lib/order-validate.ts）
  let body: AdminOrderRequest;
  try { body = await request.json(); } catch { return json({ ok: false, error_code: "INVALID_INPUT" }); }
  const invalid = validateAdminOrder(body);
  if (invalid) return json(invalid);

  const db = makeDb(env);

  // 4) Idempotency precheck FIRST — 防太太 double-click 重複建單
  // **必須**在 stock decrement 之前。若放後面，第 2 次 request 會扣 stock 然後 replay 第 1 筆，
  // 結果原 stock 沒還回去 → straight stock leak（codex F1）。
  const prior = await db.select().from(orders)
    .where(eq(orders.idempotency_key, body.idempotency_key)).limit(1);
  if (prior.length > 0) {
    return json({ ok: true, order_id: prior[0]!.order_id, total: prior[0]!.total, replayed: true });
  }

  // 5) Product + price snapshot
  const prodRows = await db.select().from(products);
  const prodMap = new Map(prodRows.map((p) => [p.sku, p]));
  let subtotal = 0;
  const itemsWithPrice = [];
  for (const it of body.items) {
    const p = prodMap.get(it.sku);
    if (!p) return json({ ok: false, error_code: "INVALID_INPUT" });
    if (!p.available) return json({ ok: false, error_code: "SOLD_OUT", sold_out_sku: it.sku });
    subtotal += p.price * it.qty;
    itemsWithPrice.push({ sku: it.sku, qty: it.qty, unit_price: p.price });
  }
  const shipping = shippingFor(body.items, env);
  const total = subtotal + shipping;

  // 6) Atomic stock reserve（precheck 已過）
  const reserve = await tryDecrementStock(env, body.items);
  if (!reserve.ok) {
    return json({ ok: false, error_code: "SOLD_OUT", sold_out_sku: reserve.sold_out_sku });
  }

  // 7) Insert（同 order_id retry pattern；複用 lib/order-errors.ts 的 isUnique* helpers）
  const idempotencyKey = body.idempotency_key;
  for (let attempt = 0; attempt < 3; attempt++) {
    const orderId = await nextOrderId(db);
    const expectedMemo = expectedMemoFor(orderId, body.name);
    const createdAt = new Date().toISOString();
    try {
      await env.DB.batch([
        env.DB.prepare(
          // pdpa_accepted=0：admin 代建單客人沒勾過 PDPA checkbox（codex F8 修正）
          `INSERT INTO orders (order_id, created_at, name, phone, address, notes, subtotal, shipping, total, expected_memo, pdpa_accepted, paid, shipped, idempotency_key)
           VALUES            (?,        ?,          ?,    ?,     ?,       ?,     ?,        ?,        ?,     ?,             0,              0,    0,       ?)`,
        ).bind(
          orderId, createdAt, body.name, body.phone, body.address,
          body.notes ?? null, subtotal, shipping, total, expectedMemo,
          idempotencyKey,
        ),
        ...itemsWithPrice.map((i) =>
          env.DB.prepare(`INSERT INTO order_items (order_id, sku, qty, unit_price) VALUES (?, ?, ?, ?)`)
            .bind(orderId, i.sku, i.qty, i.unit_price),
        ),
        env.DB.prepare(
          `INSERT INTO audit_log (ts, user_email, action, order_id, details) VALUES (?, ?, ?, ?, ?)`,
        ).bind(createdAt, session.email, "admin_order_created", orderId,
          JSON.stringify({ qty: itemsWithPrice.length })),
      ]);
      // 沒 Telegram、沒 LINE push（line_user_id 為空，且 user 不需要通知自己）
      return json({ ok: true, order_id: orderId, total });
    } catch (err) {
      if (isUniqueOnIdempotency(err)) {
        // 太太 double-click race-of-race（precheck 沒命中但 INSERT 撞 UNIQUE）
        // 把 reservation 還回去，回傳已存在的訂單
        await restoreStock(env, body.items);
        const existing = await db.select().from(orders)
          .where(eq(orders.idempotency_key, body.idempotency_key)).limit(1);
        if (!existing[0]) return json({ ok: false, error_code: "INTERNAL" });
        return json({ ok: true, order_id: existing[0].order_id, total: existing[0].total, replayed: true });
      }
      if (isUniqueOnOrderId(err)) {
        // 不要 restoreStock — reservation 跨 retry 持有
        await new Promise((r) => setTimeout(r, 50));
        continue;
      }
      // Unknown — 還 stock，bail
      await restoreStock(env, body.items);
      return json({ ok: false, error_code: "INTERNAL" });
    }
  }
  // retry 用盡
  await restoreStock(env, body.items);
  return json({ ok: false, error_code: "LOCKED" });
};
```

注意：`pdpa_accepted` 對 admin 代建單**設 0**（codex F8 修正）。原因：客人沒在 form 上勾選 PDPA checkbox，這欄位的明確意義就是「客人在 form 上勾過」。把它強設 1 等於假裝有書面同意，污染欄位語意。Taiwan PDPA 接受姑姑 LINE 上的口頭告知，但這不該由 schema 假裝。

**與 PDPA 6 個月清理的互動**：`src/cron/purge.ts` 的 `purgeOldOrders` 用 `created_at < cutoff` 過濾，**完全不看** `pdpa_accepted`。`pdpa_accepted` 只是「customer-self-accepted 紀錄」，不是 purge gate。所以 admin 單 `pdpa_accepted=0` 對 cron 沒影響，6 個月後一樣被清掉（連帶 audit_log FK cascade 刪）。

### 訂單編輯/取消的 stock 還原

V2 既有檔案：
- `src/pages/api/admin/orders/[id]/cancel.ts`（cancel endpoint）
- `src/pages/admin/orders/[id].astro`（admin 詳情頁，含取消按鈕）

改動：

- **取消訂單**（cancel.ts endpoint）：**只允許 `paid=0 AND shipped=0` 的單取消**（codex F2 + user refinement）。已付款或已出貨的單不能由這個 endpoint 處理：
  - 已付款未出貨 → 退款流程（V4 不做，V4.1+ 設計）
  - 已出貨 → 物已出，stock 不能假裝還回（會造成假庫存）
  - 未付款未出貨 → 純取消，stock 還回去給其他客人

  **兩步原子操作**——第一步 atomic UPDATE 當 gate（同時檢查 cancelled_at IS NULL + paid=0 + shipped=0），第二步 batch 包 stock restore + audit。

  ```ts
  // Step 1: atomic check-and-set as gate（含 paid + shipped 檢查）
  const result = await env.DB.prepare(
    `UPDATE orders SET cancelled_at = ?
     WHERE order_id = ? AND cancelled_at IS NULL AND paid = 0 AND shipped = 0`,
  ).bind(new Date().toISOString(), orderId).run();

  if ((result.meta.changes ?? 0) === 0) {
    // 三種情況都會走到這：已被取消、已付款、已出貨。要分辨給 user 正確錯誤訊息。
    const row = await db.select().from(orders).where(eq(orders.order_id, orderId)).limit(1);
    if (!row[0]) return json({ ok: false, error_code: "NOT_FOUND" }, 404);
    if (row[0].cancelled_at) return json({ ok: true, already_cancelled: true });
    if (row[0].paid) return json({ ok: false, error_code: "CANCEL_FORBIDDEN", reason: "paid_order_needs_refund" });
    if (row[0].shipped) return json({ ok: false, error_code: "CANCEL_FORBIDDEN", reason: "already_shipped" });
    return json({ ok: false, error_code: "CANCEL_FORBIDDEN" });
  }

  // Step 2: 只有 winner 才執行 — restore stock + audit log
  const itemsRow = await db.select().from(order_items).where(eq(order_items.order_id, orderId));
  await env.DB.batch([
    ...stockRestoreStmts(env, itemsRow),
    env.DB.prepare(`INSERT INTO audit_log (ts, user_email, action, order_id, details) VALUES (?, ?, ?, ?, ?)`)
      .bind(new Date().toISOString(), session.email, "order_cancelled", orderId, JSON.stringify({ items: itemsRow })),
  ]);
  return json({ ok: true });
  ```

  **UI 對應**：`/admin/orders/[id]` 的「取消」按鈕只在 `paid=0 && shipped=0 && cancelled_at IS NULL` 時顯示；其他狀態顯示說明文字（「已付款，需走退款流程」/「已出貨」/「已取消」）。但 server endpoint 仍要做檢查——縱深防禦。

  **不 DELETE**——保留 audit trail。誰取消這單就 `JOIN audit_log WHERE action='order_cancelled' AND order_id=?` 查。

  **Step 2 失敗的 unwind 風險**（codex F9 reword）：如果 Step 1 成功後 Step 2 失敗（D1 暫時 error / Worker isolate crash），訂單已標 cancelled 但 stock 沒 restore——資料不一致。這是設計**有意識接受**的 edge case，不是「機率夠低所以無事」。緩解：
  - admin/products 列表會顯示每個 SKU 的 stock 數字，user 對帳時可以對到不一致
  - admin 後台 stock 校正功能（PATCH endpoint）允許手動修正
  - 不加 saga / 補償 cron——對家庭規模年單量 < 1000 單，這個 edge case 預計不會發生

- **編輯訂單品項**（**全功能 item edit**——codex F5 + user F5B）：V2 既有 `edit.ts` 只改 address/notes，沒有 item-edit UI。V4 新加 `PATCH /api/admin/orders/[id]/items` endpoint + `/admin/orders/[id]` UI 改成可改完整 item 列表。

  **支援的操作**（同一個 endpoint）：
  - 改既有 line item 的 qty（diff > 0 / < 0 / = 0）
  - 新增 line item（add SKU）
  - 移除 line item（remove SKU；等同 qty = 0）
  - 同次混合（一張單同時改多個 line items）

  **限制**：只允許 `paid=0 AND shipped=0 AND cancelled_at IS NULL` 的單編輯（同 cancel 規則——已付款 / 已出貨的單不能再改）。

  **Algorithm**：

  ```
  輸入: orderId, newItems[] = [{sku, qty}, ...]
  
  1. 讀現有 line items: existing[] = [{id, sku, qty}, ...]
  2. 讀現有 order: validate paid=0 AND shipped=0 AND cancelled_at IS NULL
  3. Compute diff: 對每個 sku 計算 qtyChange = (new qty) - (existing qty)
                   - 新出現的 sku → qtyChange = +new qty (need to claim from stock)
                   - 消失的 sku    → qtyChange = -(existing qty) (give back to stock)
                   - 兩邊都有     → qtyChange = +/-/0
  4. 分流:
     - decrements = items where qtyChange > 0  (need CAS)
     - restores   = items where qtyChange < 0  (no CAS, just add back)
     - keeps      = items where qtyChange = 0
  5. Phase A: tryDecrementStock(decrements)
     若 SOLD_OUT → return 不影響任何 row
  6. Phase B: 重算 subtotal、shipping (`shippingFor`)、total
  7. Phase C: 單一 D1 batch atomic commit:
     ├ stockRestoreStmts(restores)            # 增加 stock
     ├ DELETE order_items WHERE order_id=? AND sku IN (vanished)
     ├ UPDATE order_items SET qty=? for each kept-with-diff
     ├ INSERT order_items for each new sku
     ├ UPDATE orders SET subtotal=?, shipping=?, total=?
     └ INSERT audit_log (action='order_items_edited', details=diff JSON)
  8. 若 Phase C batch 失敗:
     - 已執行 Phase A 的 decrements 必須 unwind: restoreStock(decrements)
     - throw err → return INTERNAL
  ```

  **關鍵 try/catch unwind**（同 cancel 邏輯——刻意把這條 risk window 寫進 Known Accepted Holes）：

  ```ts
  // Phase A: stock claim for additions / increments
  const decResult = await tryDecrementStock(env, decrements);
  if (!decResult.ok) {
    return json({ ok: false, error_code: "SOLD_OUT", sold_out_sku: decResult.sold_out_sku });
  }

  // Phase B: recompute money fields
  const newSubtotal = newItems.reduce((s, it) => s + prodMap.get(it.sku)!.price * it.qty, 0);
  const newShipping = shippingFor(newItems, env);
  const newTotal = newSubtotal + newShipping;

  // Phase C: atomic batch
  try {
    const vanishedSkus = existing.filter(e => !newItems.find(n => n.sku === e.sku)).map(e => e.sku);
    const newSkuItems = newItems.filter(n => !existing.find(e => e.sku === n.sku));
    const updateSkuItems = newItems.filter(n => existing.find(e => e.sku === n.sku && e.qty !== n.qty));

    await env.DB.batch([
      ...stockRestoreStmts(env, restores),
      ...(vanishedSkus.length > 0 ? [
        env.DB.prepare(`DELETE FROM order_items WHERE order_id = ? AND sku IN (${vanishedSkus.map(() => "?").join(",")})`)
          .bind(orderId, ...vanishedSkus),
      ] : []),
      ...updateSkuItems.map(it =>
        env.DB.prepare(`UPDATE order_items SET qty = ?, unit_price = ? WHERE order_id = ? AND sku = ?`)
          .bind(it.qty, prodMap.get(it.sku)!.price, orderId, it.sku)),
      ...newSkuItems.map(it =>
        env.DB.prepare(`INSERT INTO order_items (order_id, sku, qty, unit_price) VALUES (?, ?, ?, ?)`)
          .bind(orderId, it.sku, it.qty, prodMap.get(it.sku)!.price)),
      env.DB.prepare(`UPDATE orders SET subtotal = ?, shipping = ?, total = ? WHERE order_id = ?`)
        .bind(newSubtotal, newShipping, newTotal, orderId),
      env.DB.prepare(`INSERT INTO audit_log (ts, user_email, action, order_id, details) VALUES (?, ?, ?, ?, ?)`)
        .bind(now, session.email, "order_items_edited", orderId,
          JSON.stringify({ before: existing, after: newItems, decrements, restores })),
    ]);
    return json({ ok: true, total: newTotal });
  } catch (err) {
    await restoreStock(env, decrements);
    throw err;
  }
  ```

  **UI**（`/admin/orders/[id].astro`）：
  - 既有 line items 表格，每 row 含「`-` qty `+`」按鈕、刪除 X 按鈕
  - 底下「+ 新增品項」下拉，從 `available=true && stock > 0` 的 SKU 選
  - 提交時送 newItems[] 完整列表給 PATCH endpoint
  - server 算 diff，client 不做 diff 計算
  - 顯示重算後的 subtotal / shipping / total
  - paid=1 或 shipped=1 時整段 UI 是 read-only（顯示 line items 但不可改）

  **shipping recompute**：用既有 `shippingFor(items, env)` helper——V2 已有，依品項計運費。如果未來規則變複雜（如 "滿 N 包免運"），改 helper 就好，這裡不重複邏輯。

### Active orders DRY helper（新 `src/lib/orders-query.ts`）

8 個 query site 都要 filter `cancelled_at IS NULL`，重複度高。抽 helper 讓未來新加 orders query 不會忘記：

```ts
import { isNull } from "drizzle-orm";
import { orders } from "../db/schema";

/** Drizzle predicate for "active" orders (not cancelled). */
export const activeOrdersFilter = isNull(orders.cancelled_at);
```

使用方式：
- 純 select：`db.select().from(orders).where(activeOrdersFilter)`
- 帶其他條件：`db.select().from(orders).where(and(activeOrdersFilter, otherCond))`
- 純 raw SQL（如 `cron/purge.ts`）：那些寫入路徑 / 全表刪除路徑**不**用 helper（見下表「動作」欄）

新 orders query 直接用 `activeOrdersFilter`，IDE 高亮提示，幾乎不可能漏。

**`activeOrdersFilter` 過濾要加在這 8 個檔案**（已 grep 驗證）：

```ts
interface StockAdjustRequest {
  new_stock: number;       // 0 ~ 9999 整數，server 驗證
  reason: string;          // 必填，1~200 字
  override_unshipped?: boolean;  // 預設 false；若 new_stock < 未出貨 qty 加總，需明確 true 才允許
}

// Server 驗證：
// 1. new_stock 是 integer 且 0 <= new_stock <= 9999（防 typo 99999999）
// 2. reason.trim().length >= 1 && <= 200
// 3. SUM unshipped active orders qty for this sku
//    - if new_stock >= unshipped_total → 通過
//    - if new_stock < unshipped_total && !override_unshipped → 回 409 + warning + unshipped_total
//    - if new_stock < unshipped_total && override_unshipped → 通過但 audit_log details 內加 override_warning
```

1. Admin 改 stock 數字 + 填 reason → submit
2. 如果伺服器回 409 sanity warning，前端 confirm dialog「目前未出貨單共 X 包，新 stock Y 比這還少，確定？」→ confirm → 用 override_unshipped=true 重送

- `action='stock_adjusted'`
- `details=JSON.stringify({ sku, from, to, reason, override_unshipped, unshipped_total_at_adjust })`

## Known Accepted Holes

刻意不解的 case，但要寫下來避免未來忘記：

1. **手動 `wrangler d1 execute UPDATE products SET stock=N` 可能讓 stock 與未出貨單數量不一致**：例如 user 直接 SQL 把 stock 改成 5，但已有 8 個未出貨訂單佔了原本的 stock。`tryDecrementStock` 的 `WHERE stock >= ?` 只擋「當下不夠」，不會 retroactively 校驗。**接受**：手動 SQL 是緊急工具，user 自己負責；admin UI 上的「調 stock」 endpoint 會做 sanity check（看下面 Next Steps #8）。
2. **「晚上對帳」沒有專屬 UI**：premise #4 接受 fallback flow，但「對帳」這動作沒有獨立頁面——靠 user 看 `/admin` dashboard 上的「stock 數字 vs 姑姑當天 LINE 訊息」自己核對。如果未來這個摩擦變大，再做 `/admin/reconcile`。
3. **Cancel 後再 cancel**：UI 應該只在 `cancelled_at IS NULL` 時顯示「取消」按鈕，但 cancel endpoint 內仍要做 idempotent 檢查（已取消就 noop 回 200，不要重複 restoreStock 把 stock 加雙倍）。

## Success Criteria

- **客人並發測試**：兩個並發 `fetch` POST 到 `/api/orders` 搶同一個 SKU 的最後一包，**只有 1 個成功**，另一個收 `SOLD_OUT`。Stock 從 1 變 0。
- **Admin 並發測試**：兩個並發 admin 代建單請求搶最後一包，同上行為（同 helper、同 CAS 機制）。
- **Cross 並發測試**（synthetic test）：1 個客人 + 1 個 admin 同時 fetch，搶最後一包 → **只有 1 個成功**。注意這驗證的是「stock 池統一」（codex F3 reminder：實務 admin 訂單晚上才補建，**不是**即時並發；但 synthetic test 仍有價值——確認跨 endpoint 共用 stock helper 邏輯一致）。**真實 daytime cross-channel 風險靠 premise #4 fallback flow 對帳處理，不靠這條 race 防護。**
- Admin 代建單能 ship 並從 stock 正確扣減
- 改 / 取消訂單時 stock 正確還原（before/after stock 數字校驗）
- 客人前端能看到「售完」狀態（「剩 N 包」延後到 V4.1）
- 季末實測：產季結束時，admin 後台所有 SKU stock = 0，audit_log 無 `stock_below_zero` 異常記錄

**並發測試怎麼跑**：
```ts
// 在 stage 環境跑（不能用 local dev 因為 D1 isolate 行為不同）
const url = "https://mango-hsu-stage.rhsu.workers.dev/api/orders";
const body = JSON.stringify({ /* 同一個 SKU，qty=1 */ });
const [r1, r2] = await Promise.all([
  fetch(url, { method: "POST", body, headers: { /* 不同 idempotency_key */ } }),
  fetch(url, { method: "POST", body, headers: { /* 不同 idempotency_key */ } }),
]);
console.log(await r1.json(), await r2.json());
// 預期：1 個 ok:true、1 個 SOLD_OUT
```
從本機 terminal 跑 `Promise.all` 兩個 fetch，網路 latency 會讓兩請求進 worker 的時間差 ~10-50ms——對 SQLite 等級的 atomic UPDATE 來說已足夠 race。

```ts
import { spawnSync } from "node:child_process";

export function d1Execute(sql: string) {
  const r = spawnSync("wrangler", ["d1", "execute", "mango-hsu-stage", "--remote", "--command", sql], { encoding: "utf-8" });
  if (r.status !== 0) throw new Error(`wrangler d1 execute failed: ${r.stderr}`);
  return r.stdout;
}

export function seedSku(sku: string, stock: number, price = 100) {
  d1Execute(`INSERT OR REPLACE INTO products (sku, name, variant, price, available, stock, display_order) VALUES ('${sku}', 'test', 'test', ${price}, 1, ${stock}, 0)`);
}

export function cleanupTestData() {
  d1Execute(`DELETE FROM order_items WHERE order_id IN (SELECT order_id FROM orders WHERE name LIKE 'test-%'); DELETE FROM orders WHERE name LIKE 'test-%'; DELETE FROM products WHERE sku LIKE 'test-%';`);
}
```
