## Problem Statement

家裡種芒果，每年產季（約 5–9 月）都有一批家人朋友、朋友的朋友透過 LINE
私訊詢問購買芒果或芒果乾。目前接單、對帳完全靠家人手動處理 LINE 訊息，
主要痛點在**收款後的對帳**：客人個人轉帳沒有結構化的訂單備註，家人要用
肉眼比對 LINE 訊息 × 網銀交易紀錄，容易錯、容易漏、也無法重複使用。

V1 要做一個**讓現有信任圈客人下單更順、讓家人對帳更快**的網站。
關鍵機制：**訂單編號** 作為轉帳備註的強制識別碼。

## Demand Evidence

> 「每年都有人用 LINE 私訊想要購買，但用 LINE 不太好管理。」
> — 專案擁有者，2026-04-19

- **Pull 而非 push**：客人主動在產季來問
- **Recurring**：每年都發生（「每年都有人」）
- **營運痛點具體**：「收款麻煩，個人匯款無法對應訂單」

## Status Quo

- **現有通路**：LINE 私訊 → 口頭確認數量 → 口頭報帳號 → 客人轉帳 →
  家人用肉眼對 LINE 訊息 × 網銀 App 交易紀錄
- **主要痛點**：收款核對
- **其他潛在痛點**：訂單漏接、訂單資訊重複問；V1 不處理這條主線外的事

## Target User & Narrowest Wedge

**主要使用者：** 家人朋友、朋友的朋友（信任度高）

**明確排除（非 V1 目標）：**
- IG / Google 搜尋來的陌生人
- 團購主

**最小切入點：** 一個 LINE 可分享的網址 → 看芒果介紹 → 填訂單 →
拿到「訂單編號 + 轉帳備註格式 + 銀行帳號」 → 家人在 Sheet 對帳。

## Constraints

- **時間：** 芒果產季 5 月開始，V1 在 2 個週末內 ship
- **預算：** 免費或極低（Cloudflare Pages + Google Sheet 都免費；
  domain 先用 `.pages.dev`）
- **團隊：** 一人開發 + 家人後台使用（家人非工程師，介面要極簡）
- **規模：** 一季幾十 ~ 一兩百張
- **食品業法規：** 家庭小量直銷通常免登錄，但出貨前應確認 TFDA 食品業者
  登錄規定是否適用目前規模（30 秒動作，不做完會踩雷）

## Premises

使用者於 Phase 3 確認同意：

1. **P1**：使用者是信任圈，V1 不做 SEO / 品牌故事
2. **P2**：核心價值 = 結構化訂單 + 對帳輔助
3. **P3**：後端用 Google Sheet + Apps Script
4. **P4**：V1 轉帳核對是「輔助」；線上支付（LINE Pay / 綠界）是 V2
5. **P5**：季節開關必要
6. **P6**：靜態前端部署到 Cloudflare Pages
7. **P7**：一季幾十 ~ 一兩百張訂單，Sheet 足夠

## Recommended Approach — Astro + Apps Script + Google Sheet

### 技術組成
- **前端**：Astro（靜態站台）+ Tailwind CSS
- **表單**：原生 HTML form + `fetch` POST
- **後端**：Google Apps Script（bound to a dedicated Google Sheet）
- **資料**：Google Sheet（`orders` / `products` / `settings` 三個分頁）
- **部署**：Cloudflare Pages（GitHub push → auto build & deploy）

### 頁面結構（V1）
1. `/`：hero image + 一句介紹 + 當季接單狀態 + 主 CTA
2. `/products`：品種、規格、價格、季節說明、照片、**FAQ 內嵌在此頁下方**
   （不拆獨立 `/faq` 路由，省半天工）
3. `/order`：表單 + confirmation 頁

### Google Sheet schema

**分頁 `products`（V1 先寫死 placeholder，等 Open Questions 回填更新）：**
| sku | 品名 | 規格 | 單價 | available |
|---|---|---|---|---|
| AW-5 | 愛文芒果 | 5 台斤 | 650 | TRUE |
| AW-10 | 愛文芒果 | 10 台斤 | 1200 | TRUE |
| DRY-200 | 芒果乾 | 200g | 350 | TRUE |

**分頁 `settings`：**
| key | value |
|---|---|
| accepting_fresh | TRUE |
| accepting_dry | TRUE |
| shipping_flat_rate_twd | 180 |
| bank_account_display | 國泰世華 XXX-XX-XXXXXXX 戶名：○○○ |
| support_line_id | @mango-hsu |

（per-SKU 開關改成 `accepting_fresh` / `accepting_dry` 兩個 flag，
涵蓋 Open Question 5 的「鮮果 vs 果乾分開上下架」需求。）

**分頁 `orders`：**
| order_id | created_at | name | phone | items_json | subtotal | shipping | total | expected_memo | paid | shipped | tracking_no | notes |

**分頁 `errors`（webhook 失敗記錄）：**
| timestamp | error_code | request_body_json | stack_trace |

### Apps Script API 合約

**`POST /exec`** — 新增訂單
- Request body (Content-Type: `text/plain;charset=utf-8` 以避開 CORS preflight)：
  ```json
  {
    "idempotency_key": "<client-generated uuid v4>",
    "token": "<shared secret from PUBLIC_ORDER_TOKEN>",
    "honeypot": "",
    "name": "王小明",
    "phone": "0912345678",
    "address": "...",
    "items": [{"sku":"AW-5","qty":2}],
    "shipping_date_pref": "2026-06-15",
    "notes": "幫我附張卡片"
  }
  ```
- 處理：
  1. 驗 `token`；驗 `honeypot == ""`（有值直接丟）
  2. `LockService.getScriptLock()` → `waitLock(10000)`
  3. 檢查 `orders` 是否有同 `idempotency_key` 的 row；有的話直接 return 原 row
  4. 用 `Utilities.formatDate(now, "Asia/Taipei", "yyyyMMdd")` 做日期字串
  5. 讀當日最大流水號 + 1，產生 `order_id = "M-{日期}-{3位流水號}"`
  6. 讀 `products` 組價、讀 `settings` 取運費、算 total
  7. `expected_memo = "${order_id}-${name}"`
  8. Append row
  9. Release lock，return JSON
- Response：
  ```json
  { "ok": true, "order_id": "M-20260519-001", "total": 1480, "expected_memo": "M-20260519-001-王小明", "bank_account_display": "..." }
  ```
- 錯誤回應：`{ "ok": false, "error_code": "LOCKED" | "INVALID_TOKEN" | "SOLD_OUT" | "INTERNAL" }`

**`GET /exec?action=status`** — 前端載入時讀
- Response：`{ "accepting_fresh": true, "accepting_dry": true, "products":[...], "shipping": 180 }`

### CORS 處理
- Apps Script Web App **不能**自訂 CORS headers，preflight OPTIONS 不會被處理
- 前端 `fetch` 一律用 `Content-Type: text/plain;charset=utf-8`，body 是
  `JSON.stringify(payload)`，**避開 preflight**
- Apps Script 端：`JSON.parse(e.postData.contents)` 解析

### 部署流程
1. 建立一個**專用** Google 帳號（建議，避免把主帳號綁進 Web App）
2. 建立 Sheet `mango-hsu-orders`，加三個分頁（orders/products/settings）
3. Extensions → Apps Script → 貼 server code
4. Deploy → New deployment → Type: **Web app**
   - Description: `v1`
   - Execute as: **Me**
   - Who has access: **Anyone**
5. 拿到 `/exec` URL → 放進 Cloudflare Pages env var `PUBLIC_APPS_SCRIPT_URL`
6. 另外產生 `PUBLIC_ORDER_TOKEN`（32 位元隨機字串，同時寫在 Apps Script 常數
   + Pages env var；公開無所謂，只為擋無腦機器人）
7. **注意**：每次「Deploy → New deployment」會產生新 URL；修改既有部署要
   用「Manage deployments → pencil icon → New version」才能保留 URL

### 前端環境變數
```
PUBLIC_APPS_SCRIPT_URL=https://script.google.com/macros/s/.../exec
PUBLIC_ORDER_TOKEN=<shared secret>
```
（Astro `import.meta.env.PUBLIC_*` 會打包進 bundle；這是公開的 shared secret，
目的是 **raise the bar**，不是真正安全。擋無腦 scraper 夠用。）

### Duplicate-submit 保護
- 客戶端：`<button type="submit" disabled={submitting}>`，submit 時 `disabled`
- 客戶端：生成 `idempotency_key = crypto.randomUUID?.() ?? fallbackUuid()`
  （LINE in-app browser 舊版本可能沒 `crypto.randomUUID`，寫個 8 行 fallback
  用 `Date.now()` + `Math.random()` 就好）
- 客戶端 fetch 要設 `AbortController` timeout（建議 15 秒），超時顯示
  「網路不穩，請截圖並 LINE 聯絡我們」+ 顯示 support LINE ID
- Server：查到既有 `idempotency_key` 直接回傳與首次送出**相同形狀**的成功
  response（冪等）

### 對帳流程（家人視角）
- 訂單編號 = `M-20260519-001`
- Confirmation 頁寫：「**請用這行當轉帳備註：M-20260519-001-王小明**」
  + 一鍵複製按鈕 + 截圖建議
- 客人轉帳時在網銀「備註」欄貼上
- 家人看網銀 App → Ctrl+F Sheet 搜 `order_id` → `paid` 欄位打勾（checkbox 型欄）
- 出貨後填 `shipped` + `tracking_no`（家人可直接手動更新）

### 季節開關
- `settings.accepting_fresh` / `accepting_dry` 兩個 boolean
- 前端 `/` 和 `/order` 載入時呼叫 `GET /exec?action=status`
- 暫停時：首頁顯示「目前暫停接單，下一季見」；`/order` 表單不顯示對應品項
- 家人改 `settings` cell 即可，30 秒操作

## Legal & Privacy（PDPA）

- 訂單表單底部必須有**個資告知聲明**：
  > 「本網站蒐集您的姓名、電話、地址，僅用於 mango-hsu 訂單聯絡、
  > 寄送與對帳，保存至訂單完成後 6 個月後刪除。您有查詢、更正、
  > 刪除之權利，請來信 `[家人 email]`。」
- 勾選同意 checkbox 才能送出
- **自動清理**：Apps Script `time-driven trigger`（每月 1 號跑一次）刪除
  `created_at < now - 6 個月` 的 rows；避免手動遺忘
- **家人 email 佔位**：上線前必須把 `[家人 email]` 換成真實 email
  （這是 blocker，寫在 Reviewer Concerns）

## Operational Gotchas（實作前務必知道）

1. **Apps Script cold start 1–3 秒** — 客人第一次送會感覺慢，要顯示
   loading state。Mobile 在 LINE in-app browser 尤其明顯
2. **Sheet 版本備份** — 產季期間每週一次 `File → Make a copy` 存快照
3. **Apps Script 每日 quota** — UrlFetch 20k/day、execution 90 min/day；
   V1 規模遠遠用不到，但每次 webhook 失敗都要寫入 `errors` 分頁方便除錯
4. **LINE in-app browser 測試** — V1 上線前必測：iPhone LINE、Android LINE、
   Safari、Chrome 四個組合；LINE 內複製到剪貼簿的行為在某些版本有異常
5. **Edit/Cancel 流程** — V1 不做客戶端編輯；confirmation 頁註明
   「如需修改請在 LINE 聯絡我們」，家人直接在 Sheet 改

## Open Questions（BLOCKING — 寫 W1 程式前先解決 1–3）

1. **【阻塞 W1】品種、規格、單價** — 愛文、玉井、凱特、芒果乾⋯ 各幾斤裝、
   多少錢？先用 placeholder 跑，但 W1 結束前要填入真實值
2. **【阻塞 W1】運費規則** — 固定、分縣市、滿額免運？V1 先用全台固定運費
   簡化，等家人確認
3. **【阻塞 W1】冷鏈 / 常溫 / 指定到貨日** — 客人能指定到貨日嗎？表單要不要
   收日期？
4. **規模確認** — 去年實際接了幾張？若 300+，需要考慮 Sheet query 最佳化
（Open Question 5「鮮果 vs 果乾獨立開關」已在 Sheet schema 的
`accepting_fresh` / `accepting_dry` 解決。剩餘 1–4 為實際需要家人確認的項目。）

## Success Criteria

- **V1 發布時：** 網址可丟進 LINE，家人朋友能在 **3 分鐘內**完成下單
- **3 個月內：** 至少 5 張訂單透過網站送出，家人回報對帳比去年快
- **一季結束：** 訂單錯漏率低於去年

## Reviewer Concerns（review round 2 留下的非結構性議題）

Adversarial review 2nd round 評分 8/10，ship with caveats。以下項目不阻擋
動工，但 W1 或上線前要處理：

**Blockers（必須在上線前處理）：**
1. **家人 email 填進 PDPA 聲明** — 找出要用哪個信箱接個資查詢/刪除請求
2. **Open Questions 1–3** — 品種/規格/價格、運費規則、指定到貨日，
   這三個答案回來前 W1 表單只能跑 placeholder
3. **`shipping_date_pref` 欄位決策** — OQ3 若決定 V1 不收日期，要從 payload
   和表單移除；若要收，要決定日期範圍與預設值

**Polish-tier（建議但非 blocker）：**
1. 家人訓練寫成 runbook（不只口頭教學），放 `docs/family-runbook.md`
2. `PUBLIC_ORDER_TOKEN` 在 bundle 是 public secret，已承認只是 raise-the-bar；
   真的有人打爆 quota 時，V2 要加上 Cloudflare Pages Function 中繼 +
   rate limit
3. Apps Script cold start 1-3s + LockService 10s = 最差感知延遲 ~13s。
   前端 loading state 要顯示「處理中（最多 15 秒）」避免客人以為當機
4. Cloudflare Pages env var 更新後要手動 redeploy 才生效 — 寫進 runbook
5. `tracking_no` 是 ops-only 欄位；V2 可加「用 order_id 查詢訂單狀態」
   頁面給客人看

## The Assignment

**今天（或本週內）打電話給 3 個去年向你家買過芒果的 LINE 客人，問：**

> 「如果我做一個網站讓你下單，你最想要它幫你解決的一件事是什麼？」

把**原話**寫下來，貼到本文「Open Questions」最下面新開一段「客人原話」。

為什麼：你現在描述的痛點（對帳）是**家人端**的痛點。V1 編碼之前最值得
知道的是：**客人端**的痛點。他們會說什麼你現在猜不到 — 「我記不住帳號」
「每次都怕打錯數量」「我想附卡片送禮」⋯ 這通電話比寫任何 code 都值。
